### Finding 3: ╬╗ = 0 for conditional_mean is a limitation of the person-level score formula at the human MLE evaluation point

The `tune_lambda_ppi_score` formula computes:

```
╬╗* = Tr(H^-1 (C_hf + C_hf') H^-1) / (2(1+r) Tr(H^-1 V_f H^-1))
```

For the conditional-mean predictor evaluated at `human_pars`:
- `V_f` diagonal: 0.016 to 0.035 (nonzero, LLM variance is present)
- `C_hf` diagonal: **ŌłÆ0.006 to ŌłÆ0.001 (negative!)**

The cross-covariance is negative because `human_pars` overestimates discrimination for
several items (mean ratio = 1.13├Ś, ranging from 0.91├Ś to 1.40├Ś):

```
Item5:  a_true=1.257, a_human=1.758 (ratio 1.40├Ś)
Item6:  a_true=1.371, a_human=1.610 (ratio 1.17├Ś)
Item2:  a_true=0.914, a_human=1.256 (ratio 1.37├Ś)
```

When the evaluation point `human_pars` differs from `true_pars`, the score function
for conditional-mean predictions (which use `true_pars` probabilities) is evaluated
with posteriors from `human_pars`. The discrepancy creates anti-correlation between the
human score `S_h` and the conditional-mean predictor score `S_f`. The negative numerator
gets clipped to ╬╗ = 0.

**This is a limitation of the PPI++ score formula when applied at a misspecified
evaluation point.** It is not a bug; the formula is theoretically correct but requires
a good evaluation point (close to the true parameters). For the F=Y case, this limitation
is absent because `S_f = S_h` exactly regardless of the evaluation point.

For the purpose of method validation, the three-test triad should use:
- **Test A (exchangeable)**: expected ╬╗*_ppi Ōēł 0 ŌåÆ Ō£ō confirmed
- **Test B (perfect, F=Y)**: expected ╬╗*_ppi = N/(n+N) ŌåÆ Ō£ō confirmed (exactly 0.750)
- **Test B2 (ability risk, F=Y)**: expected ╬╗ > 0 ŌåÆ Ō£ō confirmed (0.8)

The conditional-mean test is degenerate for the PPI++ score formula at a misspecified
evaluation point and should be replaced in vignettes with the F=Y ability-risk test.

### Finding 4: Gradient asymmetry from frozen posteriors is the dominant cause of ╬╗ = 0 in vignette scenarios

At the human MLE starting point for the misaligned-LLM scenario, the combined gradient
at ╬╗ = 0.5 is negative (pushing a upward) for 7 out of 8 items:

```
Item  ŌłćL_obs   ŌłćL_gen  ŌłćL_pred  Combined (╬╗=0.5)
Item1 +0.0239  +0.0213  +0.0659   +0.0016   ŌåÉ barely positive
Item2 +0.0252  +0.0795  +0.1721   -0.0211   ŌåÉ negative
Item3 +0.0238  +0.0262  +0.0940   -0.0101   ŌåÉ negative
Item4 +0.0255  +0.0215  +0.1185   -0.0230   ŌåÉ negative
Item5 +0.0148  +0.0568  +0.1764   -0.0449   ŌåÉ negative (largest asymmetry)
Item6 +0.0172  +0.0234  +0.1249   -0.0336   ŌåÉ negative
Item7 +0.0228  +0.0356  +0.1235   -0.0211   ŌåÉ negative
Item8 +0.0210  +0.0062  +0.1662   -0.0590   ŌåÉ negative (largest asymmetry)
```

The asymmetry pattern `ŌłćL_pred >> ŌłćL_gen` is structurally caused by:

1. `L_pred` uses concentrated human posteriors (from `human_pars` which overestimates
   discrimination) applied to LLM responses. The concentrated posteriors create large
   residuals `N_jk * P_jk - R_jk` at the human-ability nodes.

2. `L_gen` uses LLM-specific posteriors (more diffuse because LLM discrimination is
   ~10% lower) applied to LLM responses. Diffuse posteriors give smaller residuals.

3. `ŌłćL_pred >> ŌłćL_gen` ŌåÆ combined gradient is negative ŌåÆ optimizer pushes a toward
   the false minimum at large a values.

**Consequence:** `tune_lambda_ability` selects ╬╗ = 0 because any positive ╬╗ moves
item parameters toward the false minimum, inflating the sandwich covariance and ability
risk. This is the correct mathematical outcome given the current frozen expected-count
implementation ŌĆö ╬╗ = 0 is the right answer under this objective. The wrong thing is
the objective itself.

**The fix:** Replace the frozen expected-count objective with the true marginal MML
PPI++ objective (`fit_mixed_subjects_mml`, Phase 2). This eliminates the structural
gradient asymmetry because posteriors are recomputed at each gradient evaluation,
preventing the self-consistent false minimum.

## Phase 1: Rename `tune_lambda_ability` ŌåÆ `tune_lambda_ability_risk`

### Rationale

The reviewer notes:
> "Do not use `tune_lambda_ability()` to validate PPI++ theory. It is a downstream
> scoring-risk rule, not the PPI++ oracle coefficient."

The name `tune_lambda_ability` is ambiguous ŌĆö it could be read as "tune lambda using
ability [information]" rather than "tune lambda to minimise ability risk". The rename
makes the function's objective explicit and distinguishes it from `tune_lambda_ppi_score`.

### Changes made

**`R/risk.R`:**
- Renamed primary function `tune_lambda_ability` ŌåÆ `tune_lambda_ability_risk` with full
  updated documentation. The new docstring explicitly states the minimised objective
  (E[gŌĆ▓╬Ż_╬│g], ability-score risk), distinguishes it from `tune_lambda_ppi_score`, and
  includes a diagnostic note explaining why ╬╗ = 0 is the correct mathematical outcome
  for misaligned LLMs under the frozen expected-count implementation.
- Renamed `tune_lambda_ability_crossfit` ŌåÆ `tune_lambda_ability_risk_crossfit`.
- Updated the internal call inside the crossfit function from `tune_lambda_ability` to
  `tune_lambda_ability_risk`.
- Added `.Deprecated()` wrapper for both old names: `tune_lambda_ability()` and
  `tune_lambda_ability_crossfit()`. These emit a one-time deprecation message and
  delegate to the new names. All existing code continues to work without modification.

**`NAMESPACE`:**
- Added `tune_lambda_ability_risk` and `tune_lambda_ability_risk_crossfit` exports.
- Kept `tune_lambda_ability` and `tune_lambda_ability_crossfit` exports (backward compat).

**`tests/testthat/test-core.R`:**
- Updated two test calls to the new names to remove deprecation warnings.

**Vignettes (`lambda-tuning.Rmd`, `linking-comparison.Rmd`, `mixed-subjects-workflow.Rmd`):**
- Replaced all `tune_lambda_ability(` calls with `tune_lambda_ability_risk(`.
- Replaced all `tune_lambda_ability_crossfit(` calls with `tune_lambda_ability_risk_crossfit(`.
- Replaced all backtick references (`tune_lambda_ability`) with new names.

**Test results:** 33 PASS, 0 FAIL, 0 WARN after all changes.

### Logic

The two objectives are fundamentally different:

| Function | Minimises | When to use |
|----------|-----------|-------------|
| `tune_lambda_ppi_score()` | Tr(╬Ż_╬│) ŌĆö item-parameter variance | Theoretical validation, method diagnostics |
| `tune_lambda_ability_risk()` | E[gŌĆ▓╬Ż_╬│g] ŌĆö ability-score risk | Practical calibration for operational scoring |

Using `tune_lambda_ability_risk()` for PPI++ theory validation gives misleading results
(as Phase 0 shows for the conditional_mean case). Using `tune_lambda_ppi_score()` for
practical scoring decisions ignores the propagation from item parameters to ability
estimates. The rename enforces this separation.

## Phase 2: `fit_mixed_subjects_mml` ŌĆö Marginal MML PPI++

### Why it was needed

Phase 0 confirmed the frozen expected-count gradient asymmetry is the dominant cause
of ╬╗ = 0 in misaligned-LLM scenarios. The gradient `ŌłćL_pred >> ŌłćL_gen` arises because
`L_pred` uses concentrated human-MLE posteriors while `L_gen` uses diffuse LLM-specific
posteriors. This is structural, not fixable by slope caps.

The marginal MML PPI++ objective eliminates this by recomputing posteriors at each
gradient evaluation from the current ╬│. The Bock-Aitkin identity guarantees:

    Ōłć_╬│ L^marg(╬│; Y) = expected-count gradient with posteriors from current ╬│

The true marginal likelihood has a unique minimum at the MML optimum. The false minimum
that exists in the frozen objective (where bimodal posteriors at a=4.66 are
self-consistent with the frozen expected counts) does NOT exist in the marginal
objective ŌĆö at a=4.66, the true marginal gradient for human data pushes a back down
toward the human MLE because the human responses from true a=1.26 are genuinely
inconsistent with a=4.66 in the marginal sense.

### Changes made

**`R/quadrature.R`** ŌĆö Added `posterior_and_log_lik_2pl(resp, item_pars, quadrature)`:
- Internal helper (not exported) that computes posterior weights AND per-subject
  log-normalizers log Z_i = log[╬Ż_k A_k p(Y_i|╬Ė_k; ╬│)]
- Identical logic to `posterior_weights_2pl` but saves the `logsumexp` before normalising
- Returns `list(weights = [n├Śk], log_normalizers = [n])`

**`R/internal.R`** ŌĆö Added two internal helpers:
- `marginal_loss_2pl(resp, item_pars, quadrature)`: true IRT negative marginal log-
  likelihood = ŌłÆmean(log Z_i). Differs from the frozen expected-count loss: this is
  the actual log p(Y|╬│) rather than the Q-function lower bound.
- `marginal_gradient_2pl(resp, item_pars, quadrature)`: gradient of `marginal_loss_2pl`
  via the Bock-Aitkin identity ŌĆö recomputes posteriors from current item_pars at each
  call. Verified against finite differences (max error < 1e-10).

**`R/fit.R`** ŌĆö Added `fit_mixed_subjects_mml(...)`:
- Minimises `L_obs^marg + ╬╗*(L_gen^marg ŌłÆ L_pred^marg)` where all three are true
  marginal likelihoods
- Objective and gradient share a per-call cache so posteriors are computed at most
  once per (par, dataset) combination per L-BFGS-B step
- `mml_pred_weights = "own"` (default): L_pred uses its own posteriors ŌĆö fully
  consistent objective and gradient; eliminates the frozen-posterior asymmetry
- `mml_pred_weights = "human"`: L_pred uses human posteriors (EM approximation for
  the chain rule term); retains paired-ability information
- After convergence, stores final posteriors at the converged ╬│ in `q_observed`,
  `q_predicted`, `q_generated`, so `vcov_mixed_subjects()` computes the correct
  marginal sandwich covariance without any additional function
- `slope_upper` is not required for stability (no false minima in the marginal
  objective)
- Print method shows "estimator: marginal MML PPI++"
- Exported and documented

**`R/risk.R`** ŌĆö Added `fit_fn` argument to `tune_lambda_ability_risk()`:
- Defaults to `fit_mixed_subjects` (unchanged existing behaviour)
- Pass `fit_fn = fit_mixed_subjects_mml` to use the MML estimator throughout the
  lambda grid search

**`NAMESPACE`** ŌĆö Added `fit_mixed_subjects_mml` export.

**`tests/testthat/test-core.R`** ŌĆö Added 3 new tests:
1. Gradient check: `marginal_gradient_2pl` matches finite differences to 1e-5
2. Basic MML fit: converges, returns valid object, `vcov_mixed_subjects` works
3. F=Y validation: `tune_lambda_ability_risk(..., fit_fn=fit_mixed_subjects_mml)`
   selects ╬╗ > 0 for F=Y ŌĆö the defining test of the Phase 2 fix

### Validation results

The Phase 2 implementation passes all diagnostics:

| Predictor | tune_lambda_ability_risk (frozen, slope_upper=4) | tune_lambda_ability_risk (MML) |
|-----------|--------------------------------------------------|-------------------------------|
| F=Y | **0.8** | **0.7** |
| Misaligned LLM | 0 | 0 |

**MML with misaligned LLM**:
- All convergence codes = 0 (no `slope_upper` required)
- Max discrimination values grow smoothly (1.4 to 2.4 over ╬╗ Ōłł [0,1]) instead of
  diverging to the false minimum
- Correctly selects ╬╗ = 0 (LLM is misaligned; using it does not help)
- Risk increases 3.7├Ś from ╬╗=0 to ╬╗=1 (vs 12├Ś for frozen estimator) ŌĆö problem is
  well-conditioned

**MML with F=Y (╬╗=0.7)**:
- Risk profile is U-shaped, minimum near theoretical N/(n+N) = 0.75
- The false minimum is absent: discriminations converge to MLE without slope capping

ability estimates and the risk is minimised at ╬╗ = 0. The difference from the frozen
estimator is that the risk surface is smooth and well-conditioned ŌĆö the false minimum
is gone.

### No separate `vcov_mixed_subjects_mml` needed

The existing `vcov_mixed_subjects()` works correctly for MML fits because the output
object stores posteriors at the CONVERGED parameters (not frozen initial posteriors).
The sandwich covariance computed from these posteriors is the asymptotically valid
marginal sandwich for the MML estimator.

## Phase 3: Per-item lambda

### Motivation

A global scalar ╬╗ is fragile: one subset of items with poor LLM predictions forces the
global optimum to ╬╗ = 0, preventing well-predicted items from benefiting from the LLM
data. Per-item ╬╗_j allows each item's correction weight to match the quality of the
paired LLM predictions for that item.

### Implementation

**`R/internal.R`** ŌĆö Added `item_loss_and_grad(counts, item_pars)`:
- Returns per-item contributions to the expected-count loss (n_items-vector) and
  per-item gradients (n_items-vector for a, n_items-vector for d)
- Verified: `sum(item_loss) == loss_expected_counts(...)` and
  `c(grad_a, grad_d) == gradient_expected_counts(...)` (equality up to 1e-10)
- Used by `fit_mixed_subjects_mml()` vector-lambda path

**`R/fit.R`** ŌĆö Updated `fit_mixed_subjects_mml()` to accept `lambda` as a scalar OR
length-n_items vector:
- **Scalar path (Phase 2 unchanged)**: true marginal likelihoods; posteriors recomputed
  at every gradient evaluation; fully consistent objective and gradient
- **Vector path (Phase 3 new)**: frozen expected-count objective with per-item ╬╗_j;
  posteriors frozen at `initial_pars` (same mechanism as `fit_mixed_subjects()`);
  the frozen Q-function and its gradient are fully consistent ŌĆö no code 52 convergence
  failures; `item_loss_and_grad()` applies each ╬╗_j independently to its item's
  contribution

Design rationale: the consistent-objective requirement forces the vector path to use
frozen posteriors (the Q-function lower bound). The EM approximation (gradient =
ŌłéL^marg/Ōłé╬│, not ŌłéQ/Ōłé╬│) would be inconsistent with the Q-function objective, causing
L-BFGS-B line-search failures. Per-item tuning from `tune_lambda_ability_risk_item()`
assigns ╬╗_j Ōēł 0 to items where the LLM correction is harmful, containing the
frozen-posterior gradient asymmetry to items where it is actually beneficial.

**`R/risk.R`** ŌĆö Added two new exported functions:

`tune_lambda_ppi_score_item(observed, predicted, item_pars, n_generated, ...)`:
- Applies the PPI++ Proposition 2 formula independently per item using the 2├Ś2
  diagonal block of H^{-1} for item j and the 2D sub-vectors of S_h, S_f
- Fast: O(1 posterior computation + matrix operations), no fitting required
- Returns a vector of per-item ╬╗*_j values
- Verified: F=Y gives ╬╗*_j = N/(n+N) for all j exactly Ō£ō
- Intended use: quick diagnostic for which items benefit from LLM predictions

`tune_lambda_ability_risk_item(lambda_grid, ..., init_lambda = 0, n_pass = 1)`:
- Coordinate descent: for each item j, find the ╬╗_j Ōłł `lambda_grid` that minimises
  ability-score risk while holding all other ╬╗_{j'} fixed
- `init_lambda`: starting vector (scalar broadcast to all items). **Recommended
  practice**: pass the global scalar optimum from `tune_lambda_ability_risk()` as
  `init_lambda` so the search starts near a useful operating point rather than
  all-zeros (where individual per-item improvements are hard to detect)
- `n_pass = 1`: single greedy sweep (sufficient in practice; increase for refinement)
- O(n_items ├Ś length(lambda_grid) ├Ś fitting_cost) per pass

### Recommended workflow for per-item lambda

```r
# Step 1: Find global scalar optimum
global <- tune_lambda_ability_risk(
  lambda_grid = seq(0, 1, by = 0.1),
  observed = observed, predicted = predicted, generated = generated,
  fit_fn = fit_mixed_subjects_mml, ...
)

# Step 2: Refine per-item starting from global optimum
item_tuned <- tune_lambda_ability_risk_item(
  lambda_grid = seq(0, 1, by = 0.2),
  observed = observed, predicted = predicted, generated = generated,
  init_lambda = global$best_lambda,  # start from global best
  ...
)

# Step 3: Inspect per-item lambda
print(data.frame(item = item_tuned$item, lambda = round(item_tuned$lambda, 3)))

# Step 4: Use final fit for scoring
vcov_item <- vcov_mixed_subjects(item_tuned$final_fit)
```
