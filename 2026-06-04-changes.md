### Finding 3: ╬╗ = 0 for conditional_mean is a limitation of the person-level score formula at the human MLE evaluation point

The `tune_lambda_ppi_score` formula computes:

    ╬╗* = Tr(H^-1 (C_hf + C_hf') H^-1) / (2(1+r) Tr(H^-1 V_f H^-1))

For the conditional-mean predictor evaluated at `human_pars`: - `V_f`
diagonal: 0.016 to 0.035 (nonzero, LLM variance is present) - `C_hf`
diagonal: **ŌłÆ0.006 to ŌłÆ0.001 (negative!)**

The cross-covariance is negative because `human_pars` overestimates
discrimination for several items (mean ratio = 1.13├Ś, ranging from 0.91├Ś
to 1.40├Ś):

    Item5:  a_true=1.257, a_human=1.758 (ratio 1.40├Ś)
    Item6:  a_true=1.371, a_human=1.610 (ratio 1.17├Ś)
    Item2:  a_true=0.914, a_human=1.256 (ratio 1.37├Ś)

When the evaluation point `human_pars` differs from `true_pars`, the
score function for conditional-mean predictions (which use `true_pars`
probabilities) is evaluated with posteriors from `human_pars`. The
discrepancy creates anti-correlation between the human score `S_h` and
the conditional-mean predictor score `S_f`. The negative numerator gets
clipped to ╬╗ = 0.

**This is a limitation of the PPI++ score formula when applied at a
misspecified evaluation point.** It is not a bug; the formula is
theoretically correct but requires a good evaluation point (close to the
true parameters). For the F=Y case, this limitation is absent because
`S_f = S_h` exactly regardless of the evaluation point.

For the purpose of method validation, the three-test triad should use: -
**Test A (exchangeable)**: expected ╬╗\*\_ppi Ōēł 0 ŌåÆ Ō£ō confirmed - **Test
B (perfect, F=Y)**: expected ╬╗\*\_ppi = N/(n+N) ŌåÆ Ō£ō confirmed (exactly
0.750) - **Test B2 (ability risk, F=Y)**: expected ╬╗ \> 0 ŌåÆ Ō£ō confirmed
(0.8)

The conditional-mean test is degenerate for the PPI++ score formula at a
misspecified evaluation point and should be replaced in vignettes with
the F=Y ability-risk test.

### Finding 4: Gradient asymmetry from frozen posteriors is the dominant cause of ╬╗ = 0 in vignette scenarios

At the human MLE starting point for the misaligned-LLM scenario, the
combined gradient at ╬╗ = 0.5 is negative (pushing a upward) for 7 out of
8 items:

    Item  ŌłćL_obs   ŌłćL_gen  ŌłćL_pred  Combined (╬╗=0.5)
    Item1 +0.0239  +0.0213  +0.0659   +0.0016   ŌåÉ barely positive
    Item2 +0.0252  +0.0795  +0.1721   -0.0211   ŌåÉ negative
    Item3 +0.0238  +0.0262  +0.0940   -0.0101   ŌåÉ negative
    Item4 +0.0255  +0.0215  +0.1185   -0.0230   ŌåÉ negative
    Item5 +0.0148  +0.0568  +0.1764   -0.0449   ŌåÉ negative (largest asymmetry)
    Item6 +0.0172  +0.0234  +0.1249   -0.0336   ŌåÉ negative
    Item7 +0.0228  +0.0356  +0.1235   -0.0211   ŌåÉ negative
    Item8 +0.0210  +0.0062  +0.1662   -0.0590   ŌåÉ negative (largest asymmetry)

The asymmetry pattern `ŌłćL_pred >> ŌłćL_gen` is structurally caused by:

1.  `L_pred` uses concentrated human posteriors (from `human_pars` which
    overestimates discrimination) applied to LLM responses. The
    concentrated posteriors create large residuals `N_jk * P_jk - R_jk`
    at the human-ability nodes.

2.  `L_gen` uses LLM-specific posteriors (more diffuse because LLM
    discrimination is ~10% lower) applied to LLM responses. Diffuse
    posteriors give smaller residuals.

3.  `ŌłćL_pred >> ŌłćL_gen` ŌåÆ combined gradient is negative ŌåÆ optimizer
    pushes a toward the false minimum at large a values.

**Consequence:** `tune_lambda_ability` selects ╬╗ = 0 because any
positive ╬╗ moves item parameters toward the false minimum, inflating the
sandwich covariance and ability risk. This is the correct mathematical
outcome given the current frozen expected-count implementation ŌĆö ╬╗ = 0
is the right answer under this objective. The wrong thing is the
objective itself.

**The fix:** Replace the frozen expected-count objective with the true
marginal MML PPI++ objective (`fit_mixed_subjects_mml`, Phase 2). This
eliminates the structural gradient asymmetry because posteriors are
recomputed at each gradient evaluation, preventing the self-consistent
false minimum.

## Phase 2: `fit_mixed_subjects_mml` ŌĆö Marginal MML PPI++

### Why it was needed

Phase 0 confirmed the frozen expected-count gradient asymmetry is the
dominant cause of ╬╗ = 0 in misaligned-LLM scenarios. The gradient
`ŌłćL_pred >> ŌłćL_gen` arises because `L_pred` uses concentrated human-MLE
posteriors while `L_gen` uses diffuse LLM-specific posteriors. This is
structural, not fixable by slope caps.

The marginal MML PPI++ objective eliminates this by recomputing
posteriors at each gradient evaluation from the current ╬│. The
Bock-Aitkin identity guarantees:

``` R
Ōłć_╬│ L^marg(╬│; Y) = expected-count gradient with posteriors from current ╬│
```

The true marginal likelihood has a unique minimum at the MML optimum.
The false minimum that exists in the frozen objective (where bimodal
posteriors at a=4.66 are self-consistent with the frozen expected
counts) does NOT exist in the marginal objective ŌĆö at a=4.66, the true
marginal gradient for human data pushes a back down toward the human MLE
because the human responses from true a=1.26 are genuinely inconsistent
with a=4.66 in the marginal sense.

## Phase 3: Per-item lambda

### Motivation

A global scalar ╬╗ is fragile: one subset of items with poor LLM
predictions forces the global optimum to ╬╗ = 0, preventing
well-predicted items from benefiting from the LLM data. Per-item ╬╗_j
allows each itemŌĆÖs correction weight to match the quality of the paired
LLM predictions for that item.

### Implementation

**`R/internal.R`** ŌĆö Added `item_loss_and_grad(counts, item_pars)`: -
Returns per-item contributions to the expected-count loss
(n_items-vector) and per-item gradients (n_items-vector for a,
n_items-vector for d) - Verified:
`sum(item_loss) == loss_expected_counts(...)` and
`c(grad_a, grad_d) == gradient_expected_counts(...)` (equality up to
1e-10) - Used by `fit_mixed_subjects_mml()` vector-lambda path

**`R/fit.R`** ŌĆö Updated `fit_mixed_subjects_mml()` to accept `lambda` as
a scalar OR length-n_items vector: - **Scalar path (Phase 2
unchanged)**: true marginal likelihoods; posteriors recomputed at every
gradient evaluation; fully consistent objective and gradient - **Vector
path (Phase 3 new)**: frozen expected-count objective with per-item ╬╗_j;
posteriors frozen at `initial_pars` (same mechanism as
[`fit_mixed_subjects()`](http://klintkanopka.com/mixedsubjectsirt/reference/fit_mixed_subjects.md));
the frozen Q-function and its gradient are fully consistent ŌĆö no code 52
convergence failures; `item_loss_and_grad()` applies each ╬╗_j
independently to its itemŌĆÖs contribution

Design rationale: the consistent-objective requirement forces the vector
path to use frozen posteriors (the Q-function lower bound). The EM
approximation (gradient = ŌłéL^marg/Ōłé╬│, not ŌłéQ/Ōłé╬│) would be inconsistent
with the Q-function objective, causing L-BFGS-B line-search failures.
Per-item tuning from `tune_lambda_ability_risk_item()` assigns ╬╗_j Ōēł 0
to items where the LLM correction is harmful, containing the
frozen-posterior gradient asymmetry to items where it is actually
beneficial.

**`R/risk.R`** ŌĆö Added two new exported functions:

`tune_lambda_ppi_score_item(observed, predicted, item_pars, n_generated, ...)`: -
Applies the PPI++ Proposition 2 formula independently per item using the
2├Ś2 diagonal block of H^{-1} for item j and the 2D sub-vectors of S_h,
S_f - Fast: O(1 posterior computation + matrix operations), no fitting
required - Returns a vector of per-item ╬╗\*\_j values - Verified: F=Y
gives ╬╗\*\_j = N/(n+N) for all j exactly Ō£ō - Intended use: quick
diagnostic for which items benefit from LLM predictions

`tune_lambda_ability_risk_item(lambda_grid, ..., init_lambda = 0, n_pass = 1)`: -
Coordinate descent: for each item j, find the ╬╗_j Ōłł `lambda_grid` that
minimises ability-score risk while holding all other ╬╗\_{jŌĆÖ} fixed -
`init_lambda`: starting vector (scalar broadcast to all items).
**Recommended practice**: pass the global scalar optimum from
`tune_lambda_ability_risk()` as `init_lambda` so the search starts near
a useful operating point rather than all-zeros (where individual
per-item improvements are hard to detect) - `n_pass = 1`: single greedy
sweep (sufficient in practice; increase for refinement) - O(n_items ├Ś
length(lambda_grid) ├Ś fitting_cost) per pass

## Round 2 (reviewer follow-up): Steps 1ŌĆō3

### Step 1: Correct marginal-MML sandwich covariance (`vcov_mixed_subjects_mml`)

**The problem.**
[`vcov_mixed_subjects()`](http://klintkanopka.com/mixedsubjectsirt/reference/vcov_mixed_subjects.md)
uses `avg_hessian_counts()` ŌĆö the EM complete-data Hessian ŌĆö as the
bread. For the frozen expected-count estimator this is correct. For the
marginal-MML estimator (`fit_mixed_subjects_mml`) it is not: the
complete-data Hessian over-states information by ignoring the missing
information about the latent ability ╬Ė (Louis 1982).

**The fix ŌĆö LouisŌĆÖ identity.** The correct bread for the MML estimator
is the marginal observed information:

``` R
A_marg = H_comp - I_miss
```

where `I_miss = (1/n) ╬Ż_i Var_{w_i}(s_ik)` is the average
within-posterior variance of the complete-data score vector at each
node. The meat (marginal score outer products) is already correct in the
existing implementation.

**Implementation (`R/risk.R`):**

- Added `louis_missing_info(resp, weights, item_pars)` (internal
  helper). For each pair of items (j, l), computes the three
  second-moment matrices (aa, ad, dd) via a fully vectorised
  decomposition:

  ``` R
  sm_block = 1/n ├Ś [yy - yp - t(yp) + pp]
  ```

  where `yy`, `yp`, `pp` involve outer products of response columns with
  expected- probability matrices. Cost: O(J┬▓ ├Ś n ├Ś K) ŌĆö fast (Ōēł800K ops
  for J=8, n=400, K=31). The formula was verified against numerical
  central-difference Hessian to \< 6├Ś10Ōü╗┬╣Ōü░ error across all six
  parameters.

- Added `vcov_mixed_subjects_mml(object, ridge = 1e-8)` (exported).
  Bread:
  `louis_bread(q_X) = avg_hessian_counts(q_X) - louis_missing_info(q_X)`
  for each of the three datasets, combined with the usual
  `+ ╬╗*(gen - pred)` weighting. Meat: unchanged (marginal score
  covariance).

- Updated `vcov.mixedsubjects_fit` S3 dispatch: routes to
  `vcov_mixed_subjects_mml` when
  `isTRUE(object$mml) && length(object$lambda) == 1`, and to the
  existing `vcov_mixed_subjects` otherwise.

**Empirical effect.** For the standard vignette scenario (n=400, N=1200,
F=Y): - Mean diagonal of MML sandwich: **0.0074** (Louis-corrected) -
Mean diagonal of EM sandwich: **0.0032** (over-precise)

The ~2.3├Ś larger MML uncertainty correctly accounts for the missing
information about individual abilities. The EM bread treats the latent
ability as observed, which inflates efficiency.

**Note.** The marginal Hessian is only guaranteed positive definite AT
the MLE. At the true parameters (which may differ from the sample MLE),
it can be indefinite ŌĆö this is the correct mathematical behaviour, not a
bug.

### Step 2: Fix `mml_pred_weights = "human"` objective/gradient mismatch

**The problem.** With `mml_pred_weights = "human"`, the `recompute`
closure in `fit_mixed_subjects_mml` was calling
`summarize_expected_counts(predicted, obs_r$weights)` where `obs_r` was
recomputed from the CURRENT candidate `ip` at each L-BFGS-B evaluation.
The objective changes with `ip` through both the item parameters in the
loss AND through the human posteriors used to weight the predicted term.
But the gradient only accounted for the first channel (direct gradient,
EM approximation). This mismatch meant `fn` and `gr` were inconsistent,
causing L-BFGS-B line-search failures.

**The fix.** When `mml_pred_weights = "human"`, compute the human
posteriors ONCE from `init_std` and freeze them. The resulting objective
`loss_expected_counts(pred_cts_fixed, ip)` depends on `ip` only through
the item parameters in the loss (not through the weights), so the EM
gradient `gradient_expected_counts(pred_cts_fixed, ip)` IS the gradient
of the objective. Fully consistent.

This is the **fixed-nuisance Q-function** interpretation: human
posteriors are a fixed nuisance, not a parameter.

The final `q_predicted` stored in the output also uses `pred_w_fixed` so
that `vcov_mixed_subjects_mml` uses consistent posteriors for the bread
and meat.

Added `selection_risk` column to the summary table: candidates with

### Step 3b: Cross-fit target mode and final-fit function

**`target_mode`** argument added to
`tune_lambda_ability_risk_crossfit`: - `"fixed"` (default): the full
`target_resp` is used in every fold. Correct for operational scoring
where the target population is independent of the labeled split. When
`target_resp` is not supplied, defaults to `"row_aligned"` for backward
compatibility. - `"row_aligned"`: only the training rows of
`target_resp` are used per fold. Valid when `target_resp = observed` and
fold-matched evaluation is desired.

**`final_fit_fn`** argument added to `tune_lambda_ability_risk_crossfit`
(default: `fit_mixed_subjects_split`). Allows users to pass any
final-fitting function. Note that mixing MML fold tuning with a frozen
EC final fit is an approximation; document this when reporting results.
