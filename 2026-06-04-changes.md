### Finding 3: λ = 0 for conditional_mean is a limitation of the person-level score formula at the human MLE evaluation point

The `tune_lambda_ppi_score` formula computes:

```
λ* = Tr(H^-1 (C_hf + C_hf') H^-1) / (2(1+r) Tr(H^-1 V_f H^-1))
```

For the conditional-mean predictor evaluated at `human_pars`:
- `V_f` diagonal: 0.016 to 0.035 (nonzero, LLM variance is present)
- `C_hf` diagonal: **−0.006 to −0.001 (negative!)**

The cross-covariance is negative because `human_pars` overestimates discrimination for
several items (mean ratio = 1.13×, ranging from 0.91× to 1.40×):

```
Item5:  a_true=1.257, a_human=1.758 (ratio 1.40×)
Item6:  a_true=1.371, a_human=1.610 (ratio 1.17×)
Item2:  a_true=0.914, a_human=1.256 (ratio 1.37×)
```

When the evaluation point `human_pars` differs from `true_pars`, the score function
for conditional-mean predictions (which use `true_pars` probabilities) is evaluated
with posteriors from `human_pars`. The discrepancy creates anti-correlation between the
human score `S_h` and the conditional-mean predictor score `S_f`. The negative numerator
gets clipped to λ = 0.

**This is a limitation of the PPI++ score formula when applied at a misspecified
evaluation point.** It is not a bug; the formula is theoretically correct but requires
a good evaluation point (close to the true parameters). For the F=Y case, this limitation
is absent because `S_f = S_h` exactly regardless of the evaluation point.

For the purpose of method validation, the three-test triad should use:
- **Test A (exchangeable)**: expected λ*_ppi ≈ 0 → ✓ confirmed
- **Test B (perfect, F=Y)**: expected λ*_ppi = N/(n+N) → ✓ confirmed (exactly 0.750)
- **Test B2 (ability risk, F=Y)**: expected λ > 0 → ✓ confirmed (0.8)

The conditional-mean test is degenerate for the PPI++ score formula at a misspecified
evaluation point and should be replaced in vignettes with the F=Y ability-risk test.

### Finding 4: Gradient asymmetry from frozen posteriors is the dominant cause of λ = 0 in vignette scenarios

At the human MLE starting point for the misaligned-LLM scenario, the combined gradient
at λ = 0.5 is negative (pushing a upward) for 7 out of 8 items:

```
Item  ∇L_obs   ∇L_gen  ∇L_pred  Combined (λ=0.5)
Item1 +0.0239  +0.0213  +0.0659   +0.0016   ← barely positive
Item2 +0.0252  +0.0795  +0.1721   -0.0211   ← negative
Item3 +0.0238  +0.0262  +0.0940   -0.0101   ← negative
Item4 +0.0255  +0.0215  +0.1185   -0.0230   ← negative
Item5 +0.0148  +0.0568  +0.1764   -0.0449   ← negative (largest asymmetry)
Item6 +0.0172  +0.0234  +0.1249   -0.0336   ← negative
Item7 +0.0228  +0.0356  +0.1235   -0.0211   ← negative
Item8 +0.0210  +0.0062  +0.1662   -0.0590   ← negative (largest asymmetry)
```

The asymmetry pattern `∇L_pred >> ∇L_gen` is structurally caused by:

1. `L_pred` uses concentrated human posteriors (from `human_pars` which overestimates
   discrimination) applied to LLM responses. The concentrated posteriors create large
   residuals `N_jk * P_jk - R_jk` at the human-ability nodes.

2. `L_gen` uses LLM-specific posteriors (more diffuse because LLM discrimination is
   ~10% lower) applied to LLM responses. Diffuse posteriors give smaller residuals.

3. `∇L_pred >> ∇L_gen` → combined gradient is negative → optimizer pushes a toward
   the false minimum at large a values.

**Consequence:** `tune_lambda_ability` selects λ = 0 because any positive λ moves
item parameters toward the false minimum, inflating the sandwich covariance and ability
risk. This is the correct mathematical outcome given the current frozen expected-count
implementation — λ = 0 is the right answer under this objective. The wrong thing is
the objective itself.

**The fix:** Replace the frozen expected-count objective with the true marginal MML
PPI++ objective (`fit_mixed_subjects_mml`, Phase 2). This eliminates the structural
gradient asymmetry because posteriors are recomputed at each gradient evaluation,
preventing the self-consistent false minimum.

### Logic

The two objectives are fundamentally different:

| Function | Minimises | When to use |
|----------|-----------|-------------|
| `tune_lambda_ppi_score()` | Tr(Σ_γ) — item-parameter variance | Theoretical validation, method diagnostics |

(as Phase 0 shows for the conditional_mean case). Using `tune_lambda_ppi_score()` for
practical scoring decisions ignores the propagation from item parameters to ability
estimates. The rename enforces this separation.

## Phase 2: `fit_mixed_subjects_mml` — Marginal MML PPI++

### Why it was needed

Phase 0 confirmed the frozen expected-count gradient asymmetry is the dominant cause
of λ = 0 in misaligned-LLM scenarios. The gradient `∇L_pred >> ∇L_gen` arises because
`L_pred` uses concentrated human-MLE posteriors while `L_gen` uses diffuse LLM-specific
posteriors. This is structural, not fixable by slope caps.

The marginal MML PPI++ objective eliminates this by recomputing posteriors at each
gradient evaluation from the current γ. The Bock-Aitkin identity guarantees:

    ∇_γ L^marg(γ; Y) = expected-count gradient with posteriors from current γ

The true marginal likelihood has a unique minimum at the MML optimum. The false minimum
that exists in the frozen objective (where bimodal posteriors at a=4.66 are
self-consistent with the frozen expected counts) does NOT exist in the marginal
objective — at a=4.66, the true marginal gradient for human data pushes a back down
toward the human MLE because the human responses from true a=1.26 are genuinely
inconsistent with a=4.66 in the marginal sense.

### Changes made

**`R/quadrature.R`** — Added `posterior_and_log_lik_2pl(resp, item_pars, quadrature)`:
- Internal helper (not exported) that computes posterior weights AND per-subject
  log-normalizers log Z_i = log[Σ_k A_k p(Y_i|θ_k; γ)]
- Identical logic to `posterior_weights_2pl` but saves the `logsumexp` before normalising
- Returns `list(weights = [n×k], log_normalizers = [n])`

**`R/internal.R`** — Added two internal helpers:
- `marginal_loss_2pl(resp, item_pars, quadrature)`: true IRT negative marginal log-
  likelihood = −mean(log Z_i). Differs from the frozen expected-count loss: this is
  the actual log p(Y|γ) rather than the Q-function lower bound.
- `marginal_gradient_2pl(resp, item_pars, quadrature)`: gradient of `marginal_loss_2pl`
  via the Bock-Aitkin identity — recomputes posteriors from current item_pars at each
  call. Verified against finite differences (max error < 1e-10).

**`R/fit.R`** — Added `fit_mixed_subjects_mml(...)`:
- Minimises `L_obs^marg + λ*(L_gen^marg − L_pred^marg)` where all three are true
  marginal likelihoods
- Objective and gradient share a per-call cache so posteriors are computed at most
  once per (par, dataset) combination per L-BFGS-B step
- `mml_pred_weights = "own"` (default): L_pred uses its own posteriors — fully
  consistent objective and gradient; eliminates the frozen-posterior asymmetry
- `mml_pred_weights = "human"`: L_pred uses human posteriors (EM approximation for
  the chain rule term); retains paired-ability information
- After convergence, stores final posteriors at the converged γ in `q_observed`,
  `q_predicted`, `q_generated`, so `vcov_mixed_subjects()` computes the correct
  marginal sandwich covariance without any additional function
- `slope_upper` is not required for stability (no false minima in the marginal
  objective)
- Print method shows "estimator: marginal MML PPI++"
- Exported and documented

**`R/risk.R`** — Added `fit_fn` argument to `tune_lambda_ability_risk()`:
- Defaults to `fit_mixed_subjects` (unchanged existing behaviour)
- Pass `fit_fn = fit_mixed_subjects_mml` to use the MML estimator throughout the
  lambda grid search

**`NAMESPACE`** — Added `fit_mixed_subjects_mml` export.

**`tests/testthat/test-core.R`** — Added 3 new tests:
1. Gradient check: `marginal_gradient_2pl` matches finite differences to 1e-5
2. Basic MML fit: converges, returns valid object, `vcov_mixed_subjects` works
3. F=Y validation: `tune_lambda_ability_risk(..., fit_fn=fit_mixed_subjects_mml)`
   selects λ > 0 for F=Y — the defining test of the Phase 2 fix

### Validation results

The Phase 2 implementation passes all diagnostics:

| Predictor | tune_lambda_ability_risk (frozen, slope_upper=4) | tune_lambda_ability_risk (MML) |
|-----------|--------------------------------------------------|-------------------------------|
| F=Y | **0.8** | **0.7** |
| Misaligned LLM | 0 | 0 |

**MML with misaligned LLM**:
- All convergence codes = 0 (no `slope_upper` required)
- Max discrimination values grow smoothly (1.4 to 2.4 over λ ∈ [0,1]) instead of
  diverging to the false minimum
- Correctly selects λ = 0 (LLM is misaligned; using it does not help)
- Risk increases 3.7× from λ=0 to λ=1 (vs 12× for frozen estimator) — problem is

- Risk profile is U-shaped, minimum near theoretical N/(n+N) = 0.75

ability estimates and the risk is minimised at λ = 0. The difference from the frozen
estimator is that the risk surface is smooth and well-conditioned — the false minimum
is gone.

### No separate `vcov_mixed_subjects_mml` needed

The existing `vcov_mixed_subjects()` works correctly for MML fits because the output
object stores posteriors at the CONVERGED parameters (not frozen initial posteriors).
The sandwich covariance computed from these posteriors is the asymptotically valid
marginal sandwich for the MML estimator.

## Phase 3: Per-item lambda

### Motivation

A global scalar λ is fragile: one subset of items with poor LLM predictions forces the
global optimum to λ = 0, preventing well-predicted items from benefiting from the LLM
data. Per-item λ_j allows each item's correction weight to match the quality of the
paired LLM predictions for that item.

### Implementation

**`R/internal.R`** — Added `item_loss_and_grad(counts, item_pars)`:
- Returns per-item contributions to the expected-count loss (n_items-vector) and
  per-item gradients (n_items-vector for a, n_items-vector for d)
- Verified: `sum(item_loss) == loss_expected_counts(...)` and
  `c(grad_a, grad_d) == gradient_expected_counts(...)` (equality up to 1e-10)
- Used by `fit_mixed_subjects_mml()` vector-lambda path

**`R/fit.R`** — Updated `fit_mixed_subjects_mml()` to accept `lambda` as a scalar OR
length-n_items vector:
- **Scalar path (Phase 2 unchanged)**: true marginal likelihoods; posteriors recomputed
  at every gradient evaluation; fully consistent objective and gradient
- **Vector path (Phase 3 new)**: frozen expected-count objective with per-item λ_j;
  posteriors frozen at `initial_pars` (same mechanism as `fit_mixed_subjects()`);
  the frozen Q-function and its gradient are fully consistent — no code 52 convergence
  failures; `item_loss_and_grad()` applies each λ_j independently to its item's
  contribution

Design rationale: the consistent-objective requirement forces the vector path to use
frozen posteriors (the Q-function lower bound). The EM approximation (gradient =
∂L^marg/∂γ, not ∂Q/∂γ) would be inconsistent with the Q-function objective, causing
L-BFGS-B line-search failures. Per-item tuning from `tune_lambda_ability_risk_item()`
assigns λ_j ≈ 0 to items where the LLM correction is harmful, containing the
frozen-posterior gradient asymmetry to items where it is actually beneficial.

**`R/risk.R`** — Added two new exported functions:

`tune_lambda_ppi_score_item(observed, predicted, item_pars, n_generated, ...)`:
- Applies the PPI++ Proposition 2 formula independently per item using the 2×2
  diagonal block of H^{-1} for item j and the 2D sub-vectors of S_h, S_f
- Fast: O(1 posterior computation + matrix operations), no fitting required
- Returns a vector of per-item λ*_j values
- Verified: F=Y gives λ*_j = N/(n+N) for all j exactly ✓
- Intended use: quick diagnostic for which items benefit from LLM predictions

`tune_lambda_ability_risk_item(lambda_grid, ..., init_lambda = 0, n_pass = 1)`:
- Coordinate descent: for each item j, find the λ_j ∈ `lambda_grid` that minimises
  ability-score risk while holding all other λ_{j'} fixed
- `init_lambda`: starting vector (scalar broadcast to all items). **Recommended
  practice**: pass the global scalar optimum from `tune_lambda_ability_risk()` as
  `init_lambda` so the search starts near a useful operating point rather than
  all-zeros (where individual per-item improvements are hard to detect)
- `n_pass = 1`: single greedy sweep (sufficient in practice; increase for refinement)
- O(n_items × length(lambda_grid) × fitting_cost) per pass

### Key validation results

```r
# Step 4: Use final fit for scoring

```

## Round 2 (reviewer follow-up): Steps 1–3

### Step 1: Correct marginal-MML sandwich covariance (`vcov_mixed_subjects_mml`)

**The problem.** `vcov_mixed_subjects()` uses `avg_hessian_counts()` — the EM
complete-data Hessian — as the bread. For the frozen expected-count estimator this is
correct. For the marginal-MML estimator (`fit_mixed_subjects_mml`) it is not: the
complete-data Hessian over-states information by ignoring the missing information about
the latent ability θ (Louis 1982).

**The fix — Louis' identity.** The correct bread for the MML estimator is the marginal
observed information:

    A_marg = H_comp - I_miss

where `I_miss = (1/n) Σ_i Var_{w_i}(s_ik)` is the average within-posterior variance of
the complete-data score vector at each node. The meat (marginal score outer products)
is already correct in the existing implementation.

**Implementation (`R/risk.R`):**

- Added `louis_missing_info(resp, weights, item_pars)` (internal helper).
  For each pair of items (j, l), computes the three second-moment matrices (aa, ad, dd)
  via a fully vectorised decomposition:

      sm_block = 1/n × [yy - yp - t(yp) + pp]

  where `yy`, `yp`, `pp` involve outer products of response columns with expected-
  probability matrices. Cost: O(J² × n × K) — fast (≈800K ops for J=8, n=400, K=31).
  The formula was verified against numerical central-difference Hessian to < 6×10⁻¹⁰
  error across all six parameters.

- Added `vcov_mixed_subjects_mml(object, ridge = 1e-8)` (exported).
  Bread: `louis_bread(q_X) = avg_hessian_counts(q_X) - louis_missing_info(q_X)` for
  each of the three datasets, combined with the usual `+ λ*(gen - pred)` weighting.
  Meat: unchanged (marginal score covariance).

- Updated `vcov.mixedsubjects_fit` S3 dispatch: routes to `vcov_mixed_subjects_mml`
  when `isTRUE(object$mml) && length(object$lambda) == 1`, and to the existing
  `vcov_mixed_subjects` otherwise.

**Empirical effect.** For the standard vignette scenario (n=400, N=1200, F=Y):
- Mean diagonal of MML sandwich: **0.0074** (Louis-corrected)
- Mean diagonal of EM sandwich:   **0.0032** (over-precise)

The ~2.3× larger MML uncertainty correctly accounts for the missing information about
individual abilities. The EM bread treats the latent ability as observed, which inflates
efficiency.

**Note.** The marginal Hessian is only guaranteed positive definite AT the MLE. At the
true parameters (which may differ from the sample MLE), it can be indefinite — this is
the correct mathematical behaviour, not a bug.

### Step 2: Fix `mml_pred_weights = "human"` objective/gradient mismatch

**The problem.** With `mml_pred_weights = "human"`, the `recompute` closure in
`fit_mixed_subjects_mml` was calling `summarize_expected_counts(predicted, obs_r$weights)`
where `obs_r` was recomputed from the CURRENT candidate `ip` at each L-BFGS-B
evaluation. The objective changes with `ip` through both the item parameters in the loss
AND through the human posteriors used to weight the predicted term. But the gradient
only accounted for the first channel (direct gradient, EM approximation). This
mismatch meant `fn` and `gr` were inconsistent, causing L-BFGS-B line-search failures.

**The fix.** When `mml_pred_weights = "human"`, compute the human posteriors ONCE from
`init_std` and freeze them. The resulting objective `loss_expected_counts(pred_cts_fixed, ip)`
depends on `ip` only through the item parameters in the loss (not through the weights),
so the EM gradient `gradient_expected_counts(pred_cts_fixed, ip)` IS the gradient of
the objective. Fully consistent.

This is the **fixed-nuisance Q-function** interpretation: human posteriors are a fixed
nuisance, not a parameter.

The final `q_predicted` stored in the output also uses `pred_w_fixed` so that
`vcov_mixed_subjects_mml` uses consistent posteriors for the bread and meat.

### Step 3b: Cross-fit target mode and final-fit function

**`target_mode`** argument added to `tune_lambda_ability_risk_crossfit`:
- `"fixed"` (default): the full `target_resp` is used in every fold. Correct for
  operational scoring where the target population is independent of the labeled split.
  When `target_resp` is not supplied, defaults to `"row_aligned"` for backward
  compatibility.

## Round 4 (third reviewer pass): Bug fixes

Eight issues were identified in the third static review. The reviewer confirmed

simulation results."

### Assessment

The reviewer's ranking was accurate. Issue 1 (vector-lambda quadrature summaries)
was a genuine correctness bug — if left unaddressed, `vcov()` on a per-item-lambda
fit would compute the bread and meat from posteriors that the optimizer never used,
making the sandwich covariance inconsistent with the actual estimating equation.
Issue 2 (`...` leakage) was a latent API fragility: passing `fit_fn =
fit_mixed_subjects_mml` via `...` to `tune_lambda_ability_risk_crossfit` would
silently forward that argument to `final_fit_fn`, causing an unused-argument error
or unexpected behaviour for any `final_fit_fn` that doesn't have a `fit_fn`
parameter. Issues 3–5 were documentation and test quality problems with no effect
on numerical results. The remaining three issues (1PL Louis correction, missingness
in Louis formula, EAP ability risk) were already acknowledged as deferred work.

### Fix 1: Vector-lambda quadrature summaries now store frozen posteriors (`R/fit.R`)

**Bug.** After optimization in `fit_mixed_subjects_mml()`, the output quadrature
summaries (`q_observed`, `q_predicted`, `q_generated`) were always built from the
converged parameters `conv_pars`, regardless of whether a scalar or vector lambda
was used. For the vector-lambda (frozen Q-function) path, the optimizer used
expected counts frozen at `init_std`, but the stored summaries were at `conv_pars`.
When `vcov_mixed_subjects()` was subsequently called, it computed Hessian and score
vectors from these mismatched posteriors, making the sandwich covariance inconsistent
with the estimating equation that determined `conv_pars`.

**Fix.** Added a branch on `scalar_lambda` in the final summary construction:

- **Scalar path** (unchanged): summaries built at `conv_pars`, so
  `vcov_mixed_subjects_mml()` (Louis-corrected) is consistent.
- **Vector path** (fixed): summaries built from `init_std` as item parameters and
  `obs_w_frozen`/`gen_w_frozen` as posterior weights — exactly the frozen
  posteriors that were used during optimization. `vcov_mixed_subjects()` (EM bread)
  is now consistent with the frozen Q-function.

Verified: vector-lambda `q_observed$irt_pars$a` now exactly matches `init_pars$a`.

---

### Fix 2: Explicit `fit_fn` argument in `tune_lambda_ability_risk_crossfit` (`R/risk.R`)

**Bug.** `tune_lambda_ability_risk_crossfit` had no explicit `fit_fn` parameter.
When users passed `fit_fn = fit_mixed_subjects_mml` it traveled through `...`,
which was forwarded to both the fold-level `tune_lambda_ability_risk` calls (where
it was needed) and to `final_fit_fn` via `list(...)` in `do.call` (where it caused
an unused-argument error or unexpected behaviour for functions like
`fit_mixed_subjects_split` that have no `fit_fn` parameter).

**Fix.** Added `fit_fn = fit_mixed_subjects` as an explicit named argument to
`tune_lambda_ability_risk_crossfit`. The fold-loop passes `fit_fn = fit_fn`
explicitly. The `do.call` for `final_fit_fn` no longer includes `list(...)` —
fold-tuning arguments do not contaminate the final fit call.

The consequence: any user-supplied `...` arguments (e.g. `slope_upper`) are now
forwarded to the fold tuner but NOT to the final fitter. Both are documented in
the updated `@param` block.

### Fix 3: Stale `@return` sentence in `fit_mixed_subjects_mml` (`R/fit.R`)

Updated the `@return` section to correctly describe the dispatch behaviour:

- **Scalar lambda**: `stats::vcov()` dispatches to `vcov_mixed_subjects_mml()`
  (Louis-corrected marginal sandwich). Calling `vcov_mixed_subjects()` directly
  bypasses the Louis correction.
- **Vector lambda**: summaries now store frozen posteriors; `stats::vcov()`
  dispatches to `vcov_mixed_subjects()` (EM bread), consistent with the frozen

The previous text pointed users to `vcov_mixed_subjects()` for the "correct marginal
sandwich covariance," which was wrong — the correct route is always `stats::vcov()`.

### Fix 4: MML covariance test updated to use `stats::vcov()` (`tests/testthat/test-core.R`)

The existing test for MML covariance dispatch called `vcov_mixed_subjects(fit)`,

### Fix: 1PL tuner robustness (`R/risk-1pl.R`)

Added the same `tryCatch` pattern used in `tune_lambda_ability_risk()`:

1. The `fit_fn(...)` call is wrapped; a failed fit returns `NULL` and the row gets
   `convergence = 99L` and `mean_total_risk = Inf`.
2. `vcov_mixed_subjects_1pl()` is wrapped; a failed covariance (singular bread,
   degenerate item parameters) returns `NULL` and the risk is set to `Inf`.
3. `ability_risk_1pl()` is wrapped; any remaining failure returns `Inf` risk.

The existing `selection_risk` filter then excludes all `Inf`-risk candidates before
`which.min` is called, and the existing warning fires when all candidates fail. The
behaviour is now identical to the 2PL tuner.
